# 14.1 – World Entity System

> **Project Draugr**
>
> *The world remembers every physical thing that has ever existed.*

---

# Purpose

This document defines the World Entity System used throughout Project Draugr.

The World Entity System is the foundation of the persistent world.

Every physical object that exists within the simulation is represented as a World Entity.

This system guarantees that the world never loses track of physical objects regardless of time, ownership, location, or player memory.

---

# Design Philosophy

The World Entity System follows one fundamental principle:

> **If it physically exists, it must exist in the world's registry.**

Nothing physical may exist outside the registry.

Narration is not the source of truth.

The World Entity Registry is.

---

# What is a World Entity?

A World Entity is any physical object that occupies space within the world.

Examples include:

- weapons
- tools
- clothing
- armor
- containers
- furniture
- consumables
- books
- maps
- blueprints
- documents
- raw materials (future)
- crafted objects
- decorative objects

If the Chronicle can physically touch it, move it, destroy it, repair it, steal it, or lose it—

it is a World Entity.

---

# What is NOT a World Entity?

The following are **not** World Entities:

- knowledge
- skills
- memories
- recipes
- technologies
- quests
- statistics
- player progress
- infrastructure
- weather
- wildlife populations

These are managed by other systems.

---

# Entity Identity

Every World Entity receives a permanent identifier.

Example:

```text
ITEM-000001
```

Entity IDs are:

- globally unique
- immutable
- permanent
- never reused

Destroyed entities retain their ID permanently in history.

Example:

```text
ITEM-000001

Destroyed

↓

ITEM-000002

↓

ITEM-000003
```

ITEM-000001 is never used again.

---

# World Entity Registry

The registry is the authoritative database of every physical object.

Each entity stores:

- Entity ID
- Name
- Category
- Description
- Current State
- Owner
- General Location
- Specific Location
- Parent Container
- Integrity
- Creation Date
- Last Modified
- Notes

Additional data is stored by the Item System.

---

# Entity Lifecycle

Every entity follows a lifecycle.

```text
Created
        │
        ▼
Exists
        │
        ▼
Moved
        │
        ▼
Stored
        │
        ▼
Equipped
        │
        ▼
Dropped
        │
        ▼
Recovered
        │
        ▼
Destroyed
```

Every state transition updates the registry.

---

# Entity States

Each World Entity always has one current state.

Possible states include:

- Stored
- Equipped
- Carried
- Installed
- Mounted
- Placed
- In Use
- Dropped
- Buried
- Lost
- Consumed
- Destroyed

The state represents how the object currently exists within the world.

---

# Entity Ownership

Every entity has an owner.

Possible owners include:

- Chronicle
- Settlement
- Civilization
- NPC
- Nobody

Example:

```text
Owner

Chronicle: Aldren
```

Ownership may change through:

- trade
- gifting
- theft
- inheritance
- abandonment
- death

---

# Entity Location

Every entity always has a location.

Locations are hierarchical.

Example:

```text
World

↓

Main Camp

↓

Workshop

↓

Workbench

↓

Stone Axe
```

The world always knows where every entity exists.

Objects never become "missing."

Only their location changes.

---

# Parent Container

Entities may exist inside other entities.

Example:

```text
Backpack

↓

Small Pouch

↓

Iron Needle
```

Or

```text
Storage Rack

↓

Wooden Crate

↓

Leather Boots
```

Containers create nested storage.

This allows realistic inventory management.

---

# World Persistence

Entities never disappear automatically.

If a Chronicle drops a knife inside the forest:

The knife remains there.

If another Chronicle discovers the location months later:

The same knife may still exist.

Time alone does not erase physical objects.

---

# Destruction

Entities only leave the world through destruction or consumption.

Examples:

Consumed

- food
- medicine

Destroyed

- burned
- shattered
- decomposed
- melted

Destroyed entities remain recorded historically.

---

# Registry Update Rule

Whenever any action affects a physical object:

- craft
- pick up
- equip
- unequip
- store
- transfer
- consume
- repair
- refine
- destroy
- modify

The World Entity Registry must be updated before narration is generated.

The registry is always processed first.

---

# Relationship with Other Systems

The World Entity System works alongside:

**Item System**

Defines what an entity actually is.

**Storage System**

Determines where entities are stored.

**Crafting**

Creates new entities.

**Construction**

Consumes entities.

**Infrastructure**

May contain entities.

**Chronicle System**

Owns and interacts with entities.

- provenance tracking

# Final Principle

The World Entity System is the memory of the physical world.

Chronicles may forget.

Documents may be lost.

Civilizations may collapse.

The world itself never forgets that an object once existed.
