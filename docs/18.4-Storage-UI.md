# 18.4 – Storage UI

> **Project Draugr**
>
> *Nothing exists everywhere. Everything exists somewhere.*

---

# Purpose

The Storage UI allows players to inspect the contents of physical storage objects.

Unlike traditional RPG inventories, Project Draugr has **no universal inventory**.

Every item exists as a physical object occupying a real location in the world.

The player can only inspect storage that is physically accessible.

---

# Design Philosophy

Project Draugr follows one immutable rule:

> **Storage is a physical object, not a user interface.**

The UI merely visualizes what the Chronicle is currently capable of observing.

Items do not magically follow the player.

Storage does not share contents.

Objects remain exactly where they were left until moved.

---

# No Global Inventory

Project Draugr does **not** provide:

- Infinite backpack
- Shared inventory
- Global stash
- Account storage
- Cloud inventory

Instead, players craft or acquire containers.

Examples:

- Pockets
- Small net carrier
- Sling carrier
- Backpack
- Sack
- Wooden box
- Crate
- Barrel
- Cabinet
- Shelf
- Wagon
- Cart

Every container exists independently.

---

# Opening Storage

The Storage UI only appears when the player intentionally interacts with a storage object.

Examples:

```text
Open my sling carrier.

Inspect the crate.

Open the wooden chest.

Look inside the cabinet.
```

The UI never opens automatically.

---

# Accessibility Rules

The player may only inspect storage when it is physically available.

Examples:

Allowed:

- Worn on the body
- Carried by hand
- Nearby
- Open container beside the player

Not Allowed:

- Storage left at another camp
- Buried cache miles away
- Chest inside another settlement
- Backpack forgotten in the forest

If the Chronicle cannot physically access it, the UI cannot display it.

# Capacity

Every storage object has finite capacity.

Capacity is determined by:

- internal volume
- construction quality
- physical design

Capacity is measured using real-world volume.

Example:

```text
Large Sling Carrier

Maximum Capacity

20 Liters
```

A container cannot hold objects exceeding its available volume.

---

# Weight

Storage does not ignore physics.

Every stored item contributes to the container's total weight.

Example:

```text
Container Weight

8.6 kg
```

Heavy containers become harder to:

- carry
- lift
- drag
- transport

---

# Item Ordering

Items are displayed using accumulated quantities when appropriate.

Example:

```text
🪨 Small Stone × 18
```

Unique items are displayed individually.

Example:

```text
🪓 Stone Axe

📜 Old Journal

💍 Silver Ring
```

---

# Unknown Items

The UI only displays information the Chronicle understands.

Example:

Without botanical knowledge:

```text
🍄 Unknown Mushroom
```

After learning:

```text
🍄 Red Cap Mushroom
```

A master herbalist may additionally see:

```text
🍄 Red Cap Mushroom

Edible

Best cooked before consumption.

Common near oak forests.
```

Knowledge affects description—not the item itself.

---

# Nested Storage

Containers may exist inside other containers.

Example:

```text
Backpack

├── Rope
├── Water Flask
├── Small Pouch
│   ├── Flint
│   ├── Bone Needle
│   └── Thread
└── Rabbit Meat
```

Nested storage follows the same accessibility rules.

---

# Storage Ownership

Storage does not inherently belong to anyone.

Ownership is determined socially—not mechanically.

Any accessible container may be:

- opened
- moved
- stolen
- abandoned
- destroyed

Future security systems (locks, keys, traps) modify accessibility rather than ownership.

---

# Item Identity

Every stored object exists as its own entity.

Example:

```text
Stone Axe

Item ID:
axe_004821
```

The UI does not normally display Item IDs.

They exist internally to preserve persistence.

---

# Legacy

Storage contributes directly to world history.

Examples:

A player dies.

Their backpack remains beside their body.

Future Chronicles may discover:

- the backpack
- its contents
- engraved items
- journals
- crafted tools

Nothing disappears simply because its owner died.

---

# Future Expansion

The Storage UI may later support:

- sorting
- filtering
- search
- durability display
- condition display
- container organization
- locking mechanisms
- labels
- custom names

These improvements must never violate the core principle:

Storage remains a physical object.

---

# Final Principle

The Storage UI does not represent what the player owns.

It represents what the Chronicle can physically access at that moment.

Every item exists somewhere.

Every container occupies space.

The world remembers where everything was left.
